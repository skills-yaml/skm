# CLI Design, Styling, and Output Guide (Agent Reference)

This document defines the **user experience, output formatting, and terminal UI standards** for the `skm` CLI utility. All CLI modifications MUST comply with this guide to keep the tool clean, predictable, and script-friendly.

## 1. Output Conventions

* **No unnecessary verbosity**: By default, CLI commands should output minimal, actionable information.
* **Headers**: Main actions should print high-level status headers (e.g. `Installing skills for agents...`).
* **Item Listing**: Steps or items should use lists with a single space indentation and an indicator:
  * `Linked <skill> to <path>` for operations.
  * `- <skill> (Status: <status>)` for listings.
* **Success/Error prefixes**:
  * Success messages: `[SUCCESS] <Message>` or `Successfully <action>`.
  * Failure messages: `Error: <Message>` or `[FAIL] <Message>`.

---

## 2. Stdout vs. Stderr

* **Standard Output (stdout)**:
  * Reserved for successful command output that might be piped or processed by other scripts.
  * For example, the YAML generated by configuration helpers or clean listings.
* **Standard Error (stderr)**:
  * Used for all logging, diagnostic info, loading warnings, progress messages, and error messages.
  * Ensures that redirects (e.g., `skm list > output.txt`) don't get polluted by status logs or warnings.

---

## 3. Colors and formatting

* **No raw ANSI escape codes**: Avoid hardcoding ANSI escape sequences. Add a styling dependency only when color or rich formatting is explicitly required.
* **Colors for status**:
  * Green: Successful operations or `[SUCCESS]`.
  * Red: Warnings, failures, or `[FAIL]`.
  * Yellow/Blue: Progress indicators and neutral lists.

---

## 4. Commands Help Text

* Every command and argument MUST have clear, descriptive help text defined in the `clap` derive annotations.
* Example commands MUST be documented in the command description.

---

## 5. Exit Codes

To enable safe script composition, command exit codes must be strictly respected:
* `0`: Success. The command executed completely and verified successfully.
* `1`: Operational failure (e.g., validation checks failed, a registry was unreachable, file missing).
* `2`: Command-line parsing error (managed automatically by Clap).
